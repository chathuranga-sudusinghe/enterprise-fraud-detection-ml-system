# Contributing

Thank you for your interest in contributing to the Enterprise AWS Data Lakehouse ML System. Contributions that improve the project's clarity, reliability, and usability are welcome.

## Contribution Scope

Useful contributions include:

- Documentation improvements
- Tests and test coverage improvements
- Small bug fixes
- Examples
- Setup and troubleshooting improvements
- Feature improvements that have been clearly discussed with the maintainer

## Maintainer-Led Roadmap

This project follows a maintainer-led roadmap. The maintainer sets project direction, architecture priorities, scope, and release decisions. Community ideas are welcome, but opening an issue does not guarantee that a proposed change will be accepted or scheduled.

Open or comment on an issue before beginning major work. This helps confirm that the proposal fits the project scope and avoids duplicated effort.

## Contribution Guidelines

- Keep pull requests small, focused, and reviewable.
- Do not add large or new runtime dependencies without prior maintainer approval.
- Do not commit datasets, secrets, credentials, model artifacts, or large generated files.
- Follow the existing project structure and established patterns.
- Run relevant tests and checks before opening a pull request where applicable.
- Avoid unrelated refactoring, reformatting, or file changes.
- Explain the motivation, scope, and validation performed in the pull request.

## Workflow

1. Fork the repository.
2. Create a focused branch from the appropriate base branch, using a descriptive name such as `docs/improve-setup-guide` or `fix/api-error-handling`.
3. Make the smallest change needed to address the issue.
4. Run relevant tests or checks where applicable.
5. Push the branch to your fork and open a pull request.
6. Link the related issue and complete the pull request template.

## Good First Contributions

Good first contributions may include:

- Correcting unclear or outdated documentation
- Adding a focused test for existing behavior
- Improving setup instructions for WSL2 or local development
- Adding a small usage example
- Documenting a reproducible troubleshooting solution

Look for issues labeled `good first issue` for tasks prepared for new contributors.

## Pull Request Review

Pull requests are reviewed for scope, correctness, clarity, test coverage, security, and consistency with the roadmap. Maintainers may request changes, ask that a pull request be narrowed, or close work that does not fit the project direction. Review timelines may vary, and approval is required before merging.
